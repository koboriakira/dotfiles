# 全プロジェクト共通ルール

## コア原則

- **シンプルに** — 変更はできる限りシンプルに。過剰設計しない
- **根本原因を探す** — 暫定対応ではなく本質的な解決を選ぶ
- **影響を最小に** — 必要な箇所だけ触る。関係ない変更を混ぜない

## コミュニケーションの共有ルール

- 特定の指示で上書きされないかぎり、やりとりやドキュメントは日本語で行うこと
- 指示があいまいな場合、最も合理的な解釈で進め、結果を報告する。確認のために作業を止めない
- 判断が分かれる場合のみ選択肢を提示する。明らかに優れた選択肢がある場合はそのまま実行する
- 選択肢を提示する場合は番号付き箇条書きにし、ユーザーが数字だけで回答できるようにする
  - 例: 「どちらの方針で進めますか？」→ `1. A案` / `2. B案` / `3. その他（自由入力）`

## セキュリティの共通ルール

- 本番環境の認証情報は絶対に共有しない
- 顧客データを含むファイルへのアクセスと思われる場合、かならず確認を取ること
- プロプライエタリなアルゴリズムの保護
- セキュリティクリティカルなコードの除外

## 開発の共通ルール

- 特定の指示で上書きされないかぎり、原則としてテスト駆動開発を行うこと
- すべての単機能の実装ごとに「テスト→実装→テスト実行→リファクタリング→リンター→コミット→push」のサイクルを繰り返す。テストとリンターが通れば確認なしにコミット・pushまで実行する
- 3ステップ以上の作業や設計判断が必要なタスクは、実装前にPlanモードに入る
- うまくいかなければ即座に止まって再計画する
- 振る舞いに変更がある実装の完了後、**動作確認してほしい項目のチェックリスト**をユーザーに提示すること
  - 今回の変更で何が変わったか、どう操作・実行すれば確認できるかを具体的に書く
  - 正常系だけでなく、境界条件やフォールバックの確認も含める

## 作業の定義と完了定義（Definition of Done）

**作業**とは、ユーザーの1つの指示に対する一連の対応を指す。1セッション内に複数の作業が存在しうる。

### 各作業の開始時

- [ ] タスクボードに作業を記録する

### 各作業の完了時

作業完了時に以下を確認なしに順次実行する。未達の項目があれば原因を特定して修復する。

- [ ] ユーザーの指示内容がすべて満たされている
- [ ] コード変更がある場合：テスト成功・リンターエラーなし・コミット・push済み
- [ ] タスクボードに該当セッションの作業が正しく記録されていること
- [ ] `/session-note` でObsidian のセッション記録を作成・更新した
- [ ] `/speak` でAivisSpeech による作業サマリー読み上げを実行した

### ユーザーの明示的な指示による実行

- 「メモを更新して」→ Obsidian 記録を更新する

## 学習ループ

- ユーザーから修正・指摘を受けたら、そのパターンを auto memory に記録する
- 記録する内容：何が起きたか、なぜ間違えたか、次回どうすべきか
- 同じミスを繰り返さないためのルールを自分自身で積み上げる
- auto memory は Claude Code が自動的に有用なコンテキストを保存する。`/memory` で確認・管理できる

## Slack 通知

- `/remote-control` 実行後、生成された URL を `slack-notify` コマンドで通知チャンネルに投稿する
- 通知チャンネルへの投稿が必要な場合は `slack-notify "<メッセージ>"` を使用する
